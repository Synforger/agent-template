# research/（汎用プール）

## このフォルダは何か

エージェントが user のために調査した内容を蓄積する**技術知識集約フォルダ**（汎用・横断のみ）。

WebSearch・WebFetch・各種ドキュメント読解で得た情報を、技術トピックごとに整理してまとめた成果物を置く。一度書いたら参照され続ける、**育てる前提のドキュメント群**。

## 汎用 vs プロジェクト固有（2026-05-31 改定）

- **複数プロジェクトをまたぐ汎用知識** → ここ（`research/` 直下）。例: `flowchart-best-practices.md`
- **特定プロジェクト固有の調査** → `projects/<mode>/research/`（プロジェクトを1か所開けばその調査が揃う）
- **業務固有 (= 会社プロジェクト等) の調査** → 業務リポ側 (= 親 rule 外で自己完結管理)

## 何を入れる

- ライブラリ比較・ツール選定の調査結果
- API・SDKの仕様まとめ
- アーキテクチャパターンの整理（two-stage retrieval、cooperative agent 等）
- 技術選定の検討資料（例: `rss-feed-candidates.md`）
- user が「これ調べて」と頼んだ技術調査の成果

user のスタンス: **ツール選定・ライブラリ比較は AI に任せる派**。調査の網羅性はエージェント側で担保する。

## 命名規則

- **トピック単位ファイル**: `kebab-case-name.md`
- 例: `rss-feed-candidates.md`, `vad-whisper-comparison.md`
- 1テーマ = 1ファイル。テーマが大きく派生したら別ファイルに分割

## 書き方

- `_template.md` をコピーして使う（目的・調査内容・結論・参考リンクの構成）
- フロントマター: `title`, `created`, `status`（in-progress/completed）, `tags`
- **情報源のURLは必ず残す**（後から確認・更新できるように）
- **要点は自分の言葉で書く**（コピペだけにしない）
- **結論セクションを必ず書く**（調査の意味は「で、どうすればいい?」が分かること）
- 技術選定の場合: 選択肢 + トレードオフ + 推奨案（CLAUDE.mdの提案ルールに沿った形で）

## ライフサイクル

- **読む**: 関連する話題が出た時、ピンポイントでそのファイルを開いて参照する。最初に「結論」セクションを見て、必要に応じて本文を辿る
- **作成**: user の依頼 or エージェントが必要と判断したタイミングで作る
- **更新**: 同じトピックで新情報が出たら追記・更新する（古い情報は明示的に「旧情報」として残す）
- **完了**: 調査が一段落したら `status: completed` に変える。**ファイルは削除しない**
- **finished/ や dropped/ は作らない**: researchは完了しても参照価値があるので、ステータス変更だけで管理する

## エージェントの挙動ルール

- user が何か調べてほしそうな技術話題を出したら「調べますか?」と提案する
- 調査が複数の情報源を統合する規模なら、その場の口頭回答だけで終わらせず `research/` にファイルを起こす
- 既存のresearchファイルに関連する話題が出たら、まずそのファイルを読みに行ってから話す
- 古くなりそうな情報（ライブラリ仕様・API仕様等）は、参照する前に「情報が古い可能性あり」を意識する
